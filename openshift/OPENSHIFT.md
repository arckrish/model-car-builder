# Building a Model Car in an OpenShift Cluster

Build model cars **on the cluster** with an OpenShift `BuildConfig`. No local
compute or disk needed — useful for large models and for CI/CD.

## Prerequisites

- `oc` CLI, logged in to your cluster.
- A project/namespace you can create builds and secrets in.
- A registry account (e.g. Quay) with a robot account or push token.
- A Hugging Face token for gated/private models.

## Quick start

```bash
# 1. One-time setup — creates secrets + applies the BuildConfig
export QUAY_REGISTRY=quay.io/youruser
export QUAY_USERNAME=youruser+robot
export QUAY_PASSWORD=your-quay-token
export HF_TOKEN=hf_xxxxxxxxxxxxxxxxxxxx
export NAMESPACE=model-cars          # optional

./openshift/deploy.sh setup

# 2. Trigger a build
./openshift/deploy.sh build ibm-granite/granite-3.3-2b-instruct

# Or setup + build in one step
./openshift/deploy.sh all RedHatAI/Qwen3-8B-FP8-dynamic v1.0
```

## How it works

| File | Role |
|------|------|
| [`openshift/buildconfig.yaml`](../openshift/buildconfig.yaml) | The `BuildConfig`. Contains an **inline Dockerfile** that downloads the model and shards weight files across 10 image layers for efficient push/pull. **Do not edit this file.** |
| [`openshift/deploy.sh`](../openshift/deploy.sh) | Creates the `hf-token` and `quay-push-secret` secrets, links the push secret to the `builder` service account, applies the BuildConfig, and triggers builds. |
| [`openshift/Containerfile`](../openshift/Containerfile) | A standalone copy of the buildconfig's inline Dockerfile, so you can build/test the same image locally with podman. Kept in sync manually. |
| [`openshift/pvc-model-download.yaml`](../openshift/pvc-model-download.yaml) | Alternative pattern: a `Job` + `PersistentVolumeClaim` that downloads a model to a PVC (instead of building an image). Useful for the "download once, mount many" approach. |

`deploy.sh` patches the BuildConfig's output image per build, so one
BuildConfig serves many models.

## Commands

| Command | Description |
|---------|-------------|
| `./openshift/deploy.sh setup` | Create secrets + apply the BuildConfig |
| `./openshift/deploy.sh build <repo> [tag]` | Trigger a build for one model |
| `./openshift/deploy.sh all <repo> [tag]` | Setup + build together |
| `./openshift/deploy.sh clean` | Remove the BuildConfig and secrets |

## Environment variables

| Variable | Required | Purpose |
|----------|----------|---------|
| `QUAY_REGISTRY` | yes | Registry + namespace, e.g. `quay.io/youruser` |
| `QUAY_USERNAME` | setup | Robot account / username |
| `QUAY_PASSWORD` | setup | Quay password or token |
| `HF_TOKEN` | yes | Hugging Face token |
| `NAMESPACE` | no | OpenShift project (default: current) |
| `BUILD_MEMORY` | no | Memory limit for builds (default `16Gi`) |

## Watching a build

```bash
oc get builds -n model-cars
oc logs -f build/model-car-builder-1 -n model-cars
```

## Deploy

The build pushes to `quay.io/<you>/<model-name>:<tag>`. Use it as an OCI
connection URI in OpenShift AI, or apply
[`examples/inferenceservice-modelcar.yaml`](../examples/inferenceservice-modelcar.yaml).

## Resource notes

The BuildConfig requests substantial ephemeral storage (`400Gi`) because model
downloads + sharded layers are large. Adjust to your cluster's capacity if
needed — but remember **`buildconfig.yaml` should not be edited** as part of
this project; instead override at the cluster level if you must.
